## 文档信息
- **产品名称**: ciciERP
- **版本**: v1.0
- **文档日期**: 2026-02-26
- **产品定位**: 多渠道销售平台的统一后台管理系统

---

## 1. 产品概述

### 1.1 产品定位

ciciERP 是一个独立的 ERP 后台管理系统，作为多个前端销售平台的公共后台，提供统一的产品、客户、供应商、物流、采购和订单管理能力。

**核心价值主张**：
- **统一管理**: 一个后台管理多个销售渠道，避免数据分散
- **智能助手**: 内置 AI 管家，通过飞书实现自然语言交互
- **灵活扩展**: 支持多平台接入，模块化设计
- **数据驱动**: 完整的业务数据链，支持决策分析

### 1.2 应用场景

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   cicishop      │     │   其他平台       │     │   线下渠道       │
│   (独立站)       │     │   (Amazon/等)   │     │   (批发/门店)    │
└────────┬────────┘     └────────┬────────┘     └────────┬────────┘
         │                       │                       │
         └───────────────────────┼───────────────────────┘
                                 │
                    ┌────────────▼────────────┐
                    │      ciciERP API        │
                    │   (统一数据同步接口)      │
                    └────────────┬────────────┘
                                 │
         ┌───────────────────────┼───────────────────────┐
         │                       │                       │
┌────────▼────────┐    ┌────────▼────────┐    ┌────────▼────────┐
│  产品管理(PIM)  │    │  订单管理(OMS)  │    │  AI管家(飞书)   │
```

## 3. ciciERP AI 管家

### 3.1 设计理念

ciciERP AI 管家是基于 Claude 的人工智能助手，通过飞书与用户交互，用自然语言完成日常 ERP 管理任务。

**核心价值**：
- **零学习成本**: 无需学习复杂系统，自然语言即可操作
- **提升效率**: 常规任务自动化，减少人工操作
- **智能预警**: 主动发现问题并提醒
- **随时随地**: 通过飞书移动端随时管理

### 3.2 交互方式

#### 3.2.1 支持的交互形式

| 形式 | 说明 | 示例 |
|-----|------|------|
| **文字指令** | 文字对话 | "帮我查询今天的订单" |
| **语音指令** | 语音转文字 | 发送语音消息 |
| **图片识别** | 识别产品/单据 | 发送产品图片询价 |
| **快捷按钮** | 预设常用指令 | 点击"今日待办"按钮 |

#### 3.2.2 对话模式

**1. 任务执行模式**
```
用户: 帮我创建一个采购订单，从供应商A采购100个产品X
AI管家: 好的，我来为您创建采购订单。
      [执行创建操作]
      AI管家: 已成功创建采购单 PO20260226001，供应商：A公司，产品：产品X，数量：100，总额：¥5,000。
```

**2. 数据查询模式**
```

AI管家: 今天共有 28 个新订单，其中：
      - cicishop: 23单
```

**3. 智能提醒模式**

## 4. 平台接口设计

### 4.1 接口架构

#### 4.1.1 API 设计原则

| 原则 | 说明 |
|-----|------|
| **RESTful** | 遵循 REST 架构风格 |
| **版本控制** | URL 包含版本号 (如 /api/v1/) |
| **统一响应** | 统一的响应数据格式 |
| **幂等性** | 相同操作多次执行结果一致 |
| **安全性** | API Key 认证 + 请求签名 |
| **限流** | 防止恶意请求，保证服务稳定 |

#### 4.1.2 通用响应格式

```json
{
  "code": 200,
  "message": "success",
  "data": {},
  "timestamp": 1707264000,

}
```

#### 4.3.1 同步方向

| 数据类型 | 同步方向 | 频率 | 说明 |
|---------|---------|------|------|
| **产品数据** | ERP → 平台 | 实时/定时 | ERP为主，平台同步 |
| **库存数据** | ERP → 平台 | 实时 | ERP为主，平台同步 |
| **订单数据** | 平台 → ERP | 实时 | 平台为主，ERP同步 |
| **客户数据** | 平台 → ERP | 实时 | 平台为主，ERP同步 |

#### 4.3.2 事件驱动架构

```

```

**事件类型**:
- `order.created` - 订单创建
- `order.paid` - 订单付款
- `order.cancelled` - 订单取消
- `product.updated` - 产品更新
- `inventory.changed` - 库存变更

### 4.4 幂等性保证

所有写操作接口支持幂等性：

1. **幂等键**: 请求头包含 `Idempotency-Key`
2. **去重处理**: 相同幂等键的请求，返回第一次执行结果
3. **过期机制**: 幂等键记录24小时后过期

### 4.5 异常处理

#### 4.5.2 重试机制

| 错误类型 | 重试策略 | 说明 |

## 6. 核心数据字典

### 6.1 数据实体关系

```
customers (客户)
  ├── orders (订单)
  │   ├── order_items (订单明细)
  │   └── shipments (发货单)
  ├── customer_addresses (客户地址)
  └── customer_tags (客户标签)

products (产品)
  ├── product_skus (SKU)
  └── product_images (产品图片)

suppliers (供应商)
  ├── purchase_orders (采购单)
  │   └── purchase_order_items (采购明细)
  └── supplier_products (供应商产品)

inventory (库存)
  ├── stock_movements (库存流水)
  └── stock_alerts (库存预警)

logistics (物流)
  ├── logistics_companies (物流公司)
  └── shipping_templates (运费模板)
```

### 6.2 核心枚举值

#### 订单状态 (order_status)
```
1 - 待审核
2 - 待发货
3 - 部分发货
4 - 已发货
5 - 已完成
6 - 已取消
7 - 售后处理中
```

#### 付款状态 (payment_status)
```
1 - 未付款
2 - 部分付款
3 - 已付款
4 - 已退款
5 - 部分退款
```

#### 发货状态 (fulfillment_status)
```
1 - 未发货
2 - 部分发货
3 - 已发货
4 - 已签收
```

#### 产品状态 (product_status)
```
1 - 上架
2 - 下架
3 - 售罄
4 - 删除
```

#### 客户等级 (customer_level)
```
1 - 普通客户
2 - 银卡客户
3 - 金卡客户
4 - 钻石客户
5 - 批发客户
```

#### 供应商评级 (supplier_rating)
```
A - 优秀 (评分≥4.5)
B - 良好 (评分≥4.0)
C - 一般 (评分≥3.5)
D - 较差 (评分<3.5)
```
