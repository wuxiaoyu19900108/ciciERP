# ciciERP 架构与页面评审报告

## 评审信息

| 项目 | 内容 |
| --- | --- |
| **评审日期** | 2026-04-12 |
| **评审范围** | `PRODUCT_SPEC.md`、`ARCHITECTURE.md`、Rust workspace 实际代码、`https://erp.cicishop.cc` 线上页面 |
| **评审方式** | 文档对照、代码审查、`cargo test` / `cargo build` 基线检查、Playwright 只读巡检 |
| **重点** | 架构合理性、模块边界、跨模块冲突、cicishop 对接可用性、AI 管家设计、页面与 CRUD 一致性 |

## 执行摘要

| 维度 | 评分（1-5） | 结论 |
| --- | --- | --- |
| **架构合理性** | **4.0** | Axum + SQLite + 分层 crate 对小团队很合适，整体方向正确 |
| **模块边界清晰度** | **3.5** | `api/db/models/utils` 边界基本清楚，但 Web 层过度集中 |
| **模块间冲突控制** | **2.5** | 已出现真实路由冲突、列表/详情不一致、UI/API 能力不对齐 |
| **cicishop 兼容性** | **2.0** | 集成鉴权方案合理，但关键对接接口仍未完成 |
| **AI 管家设计** | **2.0** | 文档设计完整，代码层基本未落地，当前只能算规划 |
| **线上页面一致性** | **2.5** | 客户/供应商/订单较稳，产品/采购/物流/统计存在明显问题 |
| **综合评分** | **2.8** | **底层架构可继续演进，但当前实现离“稳定可交付”还有一段距离** |

**总体判断**：项目的底层技术路线是合理的，尤其适合当前这种轻量 ERP 场景；真正的问题不在“选型错了”，而在**实现完成度、文档漂移、Web 路由一致性和跨页面闭环**。目前最需要修的是线上已暴露的问题和 cicishop 集成缺口，而不是再做一轮大重构。

## 1. 架构合理性评审

### 1.1 总体架构

| 项目 | 现状 | 评价 | 评分 |
| --- | --- | --- | --- |
| **后端结构** | `cicierp-api -> cicierp-db / cicierp-models / cicierp-utils` | 分层清晰，无明显循环依赖 | 4.5 |
| **运行时与 Web 框架** | Axum + Tokio | 适合 SSR + API 一体化，复杂度可控 | 4.0 |
| **数据层** | SQLite + WAL + sqlx | 对当前体量务实，部署成本低 | 4.0 |
| **鉴权分层** | 普通 JWT 鉴权与 integration HMAC 鉴权分开 | 思路正确，边界清楚 | 4.0 |
| **Web 页面层** | `crates/api/src/routes/web.rs` 单文件约 9543 行 | 页面、模板、业务逻辑过度耦合，维护风险高 | 2.0 |

**结论**：项目不是“架构崩了”，而是**底层架构可用，表现层实现开始失控**。尤其是 `web.rs` 和 `web_invoice.rs` 把多个模块的页面、HTML 拼接、查询与业务判断塞在一起，后续越改越容易出现跨模块回归。

### 1.2 技术选型是否符合产品需求

| 需求 | 当前选型 | 结论 |
| --- | --- | --- |
| 中小团队 ERP、单体部署 | Rust + SQLite | **匹配** |
| SSR 后台管理 | Axum + HTML 拼接 / Tailwind | **能用，但维护成本偏高** |
| 多平台对接（cicishop 等） | 独立 integration middleware + route | **方向正确** |
| AI 管家 | 文档有设计，代码未实现 | **当前不匹配** |
| 数据分析看板 | 页面已存在，数据闭环不足 | **部分匹配** |

## 2. 文档与代码的一致性

| 文档说法 | 实际代码 | 结论 |
| --- | --- | --- |
| `ARCHITECTURE.md` 有 `frontend/`、Leptos WASM 前端 | workspace 中没有 Leptos crate，也没有 `frontend/` | **文档过时** |
| `ARCHITECTURE.md` 有 `ai-butler/`、飞书 Bot、Claude API | `crates/` 下无相关实现 | **文档超前于代码** |
| `README.md` / `.github/copilot-instructions.md` 标记 AI 管家、analytics dashboard 为 pending | 与实际代码更接近 | **README 比架构文档更真实** |
| 文档强调 AI 管家是核心组成 | 当前线上与代码中均无 AI 入口 | **产品承诺与交付状态不一致** |

**结论**：当前最大的问题不是“代码背离设计”，而是**文档把未来规划写成了当前架构**。这会直接误导后续开发、测试和对外沟通。

## 3. 模块冲突与逻辑一致性

### 3.1 已确认的冲突/断裂点

| 问题 | 证据 | 影响 | 优先级 |
| --- | --- | --- | --- |
| **采购路由命名不一致** | Web 路由实际使用 `/purchase`、`/purchase/new`；仪表板快捷入口却跳 `/purchases/new` | 线上快捷入口直接 404 | **P0** |
| **产品列表与详情不一致** | Playwright 在 `/products` 取到 `SP-147 -> /products/189`，进入详情页显示“产品不存在” | 用户无法从列表进入真实详情，编辑页也随之失效 | **P0** |
| **物流页面“查看”无实际跳转价值** | 线上 `/logistics` 页面多个“查看”链接仍指向 `/logistics` 自身 | 用户认知上像有详情页，实际没有 | **P1** |
| **库存模块页面能力定义不统一** | 列表页有“调整/详情/流水/删除”，但无“新增库存”入口；`/inventory/new` 实际可访问 | 页面导航与功能存在脱节 | **P1** |
| **采购模块 CRUD 不完整** | 线上有列表/新建/详情，但 `/purchase/:id/edit` 不存在；代码也未注册编辑页 | 与其他主数据模块的交互模型不一致 | **P1** |
| **统计口径容易形成冲突感知** | 仪表板显示总订单/待发货有值，但分析页默认展示“暂无销售数据” | 用户会怀疑统计错误，需要明确筛选口径 | **P2** |

### 3.2 代码层耦合风险

| 风险点 | 说明 | 影响 |
| --- | --- | --- |
| **`web.rs` 过大** | 页面路由、HTML 拼接、查询和交互逻辑集中在单文件 | 任一模块改动都可能波及别的模块 |
| **Web 路径命名未统一** | `/purchase` 单数、API `/purchases` 复数并存 | 容易再出现新的断链 |
| **模块 UI 完成度不均衡** | 客户/供应商已形成完整闭环，物流/采购/库存则各走各的交互模型 | 系统体验不一致 |

## 4. cicishop 兼容性评审

### 4.1 正面评价

| 项目 | 评价 |
| --- | --- |
| **对接鉴权模型** | `integration_auth_middleware` 采用 API Key + HMAC + 时间戳，设计合理 |
| **接口边界** | 对接 API 独立挂在 `/api/v1/integration`，与后台 API 隔离清楚 |
| **权限模型** | `IntegrationClient` 按 `products:*` / `orders:*` 等权限粒度控制，方向正确 |

### 4.2 阻塞问题

| 接口/能力 | 当前状态 | 结论 | 优先级 |
| --- | --- | --- | --- |
| `POST /api/v1/integration/orders` | TODO，直接返回 Not implemented | 无法完成商城订单导入 | **P0** |
| `PUT /api/v1/integration/orders/:platform_order_id` | TODO | 无法回写/更新平台订单 | **P0** |
| `GET /api/v1/integration/inventory` | TODO，占位返回空分页 | 库存同步结果不可信 | **P0** |
| `GET /api/v1/integration/inventory/sku/:sku_code` | TODO | 无法按 SKU 精准查询库存 | **P0** |
| `POST /api/v1/integration/inventory/reserve` | TODO | 预留库存流程不可用 | **P0** |
| `POST /api/v1/integration/inventory/release` | TODO | 释放库存流程不可用 | **P0** |
| 客户 external_id 查询/批量同步 | TODO | 客户映射闭环不完整 | **P1** |

### 4.3 安全与稳定性问题

| 问题 | 位置 | 影响 | 优先级 |
| --- | --- | --- | --- |
| **请求体读取无上限** | `crates/api/src/middleware/integration_auth.rs` 使用 `to_bytes(body, usize::MAX)` | 大包可直接拉高内存风险 | **P0** |
| **幂等键定义了但未真正落地** | `integration.rs` 多个请求结构包含 `idempotency_key`，处理逻辑未实现 | cicishop 重试时可能产生重复写入 | **P0** |

**结论**：与 cicishop 的**接口框架已经搭好**，但**关键业务操作还没有落地**。当前状态不适合把对接能力当成“已可用”对外承诺。

## 5. AI 管家设计评审

| 评审项 | 现状 | 评分 |
| --- | --- | --- |
| **飞书集成方案** | 文档有模块划分和交互思路，代码无实现 | 2.0 |
| **意图处理设计** | 文档层面较完整，但没有实际任务路由、工具编排、回退策略实现 | 2.0 |
| **权限控制** | 主系统已有用户/角色/权限基础，可复用；AI 侧未接入 | 2.5 |
| **安全性** | 缺飞书用户映射、审计日志、敏感指令保护、额度控制 | 2.0 |

**结论**：AI 管家目前仍是**设计稿阶段**。如果后续要做，建议按这个顺序落地：

1. 飞书用户与 ERP 用户映射。
2. 只读类查询意图。
3. 审批型动作。
4. 真正写入型业务动作。

不要一开始就让 AI 直接做高风险写操作。

## 7. 主要问题与改进建议

| 优先级 | 问题 | 建议 |
| --- | --- | --- |
| **P0** | 采购快捷入口路径错误 | 全量统一采购 Web 路径命名；至少先修复仪表板快捷入口为 `/purchase/new` |
| **P0** | 产品列表到详情存在脏链接/取数错位 | 排查产品列表生成链接与详情查询条件，优先修复线上不可达详情 |
| **P0** | cicishop 集成核心接口未实现 | 先补齐订单创建/更新、库存查询、预留、释放；再谈平台联调 |
| **P0** | integration 鉴权 body 无限读取 | 给请求体加明确上限，并补充超限错误响应 |
| **P0** | 幂等键未真正使用 | 为订单/库存接口落地幂等表或缓存记录，避免重试重复写入 |
| **P1** | `web.rs` 过大、页面耦合严重 | 按模块拆分为 `web_products.rs`、`web_orders.rs`、`web_customers.rs` 等 |
| **P1** | 物流 Web UI 缺少真正详情页 | 要么补详情页/编辑页，要么把“查看”改成可展开信息，避免误导 |
| **P1** | 采购与库存模块交互模型不统一 | 明确哪些模块支持完整 CRUD，哪些模块走“调整/审批”式流程，并在 UI 上体现 |
| **P1** | 架构文档严重超前 | 把 `ARCHITECTURE.md` 改成“当前实现 + 规划项”两层描述，不要把未落地内容写成现状 |
| **P2** | 构建 warning 较多 | 清理 `unused imports`、`ambiguous_glob_reexports` 等噪音，降低后续回归排查成本 |
| **P2** | 统计口径表达不清 | 在仪表板和分析页明确“本月 / 当前筛选 / 全量”口径，减少误判 |

## 8. 建议开发顺序

1. **先修线上真实缺陷**：采购快捷入口、产品详情断链、物流误导性“查看”。
2. **再补平台对接闭环**：订单导入、库存预留/释放、external_id 客户映射、幂等控制。
3. **然后处理 Web 层可维护性**：拆分 `web.rs`，统一模块路由规范。
4. **最后更新文档与 AI 规划**：把架构文档收敛为真实状态，再列后续版本路线。

## 9. 最终结论

**这套系统的底座是能成立的**：Rust 单体 + SQLite + 分层 crate 对当前阶段是合理选择，没有必要推倒重来。

**但当前最大短板不是架构，而是完成度和一致性**：

1. cicishop 对接能力还没真正闭环。
2. 线上页面已经暴露出路径和数据一致性问题。
3. 文档把未来规划写成了现在实现，容易误导。

如果按优先级逐项修复，项目是可以继续稳定迭代的；如果不先收敛这些基础问题，后续再叠加 AI、统计、对接功能，只会把现有不一致放大。
